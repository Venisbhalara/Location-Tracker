Feature Requirement:

1. User Access Control:

* When a user wants to use the location tracking feature, they should NOT get direct access.
* Instead, they must first submit an **access request** to the admin.

2. Admin Approval Workflow:

* Admin should have a dedicated **Admin Panel Dashboard**.

* Admin can view all pending requests.

* Each request should include:

  * User details (name, phone number, or identifier)
  * Request timestamp

* Admin can:

  * ✅ Approve the request → user gets access to tracking feature
  * ❌ Reject the request → user is denied access

3. Access Enforcement:

* Only **approved users** can:

  * Generate tracking links
  * Send or receive live location

* Rejected or pending users should:

  * See a “Waiting for Admin Approval” or “Access Denied” message
  * Be blocked from using tracking APIs

4. Backend Requirements:

* Create a database table (e.g., `access_requests`) with fields:

  * id
  * user_identifier (phone/email)
  * status (pending / approved / rejected)
  * created_at
  * updated_at

* Create APIs:

  * POST /request-access → user sends request
  * GET /admin/requests → admin fetches all requests
  * PUT /admin/approve/:id → approve request
  * PUT /admin/reject/:id → reject request

5. Middleware Protection:

* Implement middleware to:

  * Check if user is approved before allowing access to tracking routes
  * Block unauthorized users

6. Frontend Requirements:

* User side:

  * Button: “Request Access”
  * Status display (Pending / Approved / Rejected)

* Admin side:

  * Dashboard showing all requests
  * Approve / Reject buttons

7. Security & Best Practices:

* Admin routes should be protected (basic auth or JWT)
* Prevent unauthorized users from accessing admin panel
* Ensure scalable and clean architecture

Goal:
Build a complete production-ready admin-controlled access system where only approved users can use the location tracking feature, ensuring security and controlled usage.
